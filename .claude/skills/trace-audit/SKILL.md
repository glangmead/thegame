# Trace Audit

Audit a `.game.jsonc` game's behavior by comparing a play trace against
the original rulebook. The trace was generated by the gamer CLI tool.

## Isolation

The audit agent MUST run in a temporary directory containing only:
- The trace file(s)
- The rulebook (PDF or rules-sentences JSON)
- Card data JSON (if applicable)

No game source code, no `.game.jsonc`, no test files. This ensures the
agent's interpretation of the rules is independent from the implementation.

To set up the isolation directory:
1. Create a temporary directory (e.g., `/tmp/trace-audit-<game>/`)
2. Copy the trace file(s) into it
3. Copy the rulebook PDF and/or rules-sentences JSON into it
4. Copy card data JSON if the game has cards
5. Spawn the audit agent with that directory as its working directory.
   `isolation: "worktree"` is NOT appropriate here since the audit
   agent should not see the repo at all. Instead, use a plain temp
   directory and pass file paths to the agent.

Example invocation from the main agent:

```
# Set up isolation directory
mkdir -p /tmp/trace-audit-lod
cp /tmp/traces/trace_LegionsOfDarkness_*.txt /tmp/trace-audit-lod/
cp pdfs/Legions_of_Darkness_ocr.pdf /tmp/trace-audit-lod/
cp nocommit/docs/lod_rules_audit.json /tmp/trace-audit-lod/
cp nocommit/docs/lod_cards.json /tmp/trace-audit-lod/  # if exists

# Spawn audit agent
Agent(
  prompt: "Audit the game traces in /tmp/trace-audit-lod/ against
    the rulebook. Read the skill instructions first.",
  subagent_type: "general-purpose"
)
```

## Process

For each trace file:

1. Read the rulebook cover-to-cover first. Build a mental model of the
   game's phases, actions, and state transitions.

2. Read the trace step by step. For each step, verify:

   **Offered actions**: Given the state shown (or reconstructed from
   step 0 + cumulative changes), are all listed actions correct per the
   rules? Are any missing? Are any wrongly included?

   **State changes (CHANGED)**: After the chosen action, do the changes
   match the rules? Check:
   - Arithmetic (DRM sums, retreat distances, resource costs)
   - Conditional logic (when effects apply or don't)
   - Completeness (no missing side effects)
   - Phase transitions

   **Logs**: Do the log messages match the state changes? Inconsistency
   between logs and CHANGED lines may indicate a bug in either.

3. Output a numbered list of findings grouped by severity:
   - **CRITICAL**: Affects game outcome (wrong combat, missing victory
     check, incorrect cost)
   - **MODERATE**: Affects strategy (wrong availability, missing option)
   - **LOW**: Cosmetic or edge-case only

   For each finding, cite the rule number and explain what the trace did
   vs. what the rules say.

## Important

- You should only have access to trace files, the rulebook, and card data.
  If you can see game source code or `.game.jsonc` files, the isolation
  setup is wrong -- flag this to the user immediately.
- Do NOT assume the trace is correct. Question everything.
- Focus on steps where state actually changes (non-empty CHANGED block).
  Auto-picked phase transitions are usually correct and can be skimmed.
- Card effects are often the source of bugs. Pay extra attention to
  event resolution steps.
